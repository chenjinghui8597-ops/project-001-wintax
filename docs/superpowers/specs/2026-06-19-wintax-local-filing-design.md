## 1. 背景与目标

本项目目标是制作一款本地 Windows 桌面软件，用于管理出口退税相关的数字化单证备案资料。

参考视频中的“中天易税云平台——数字化单证备案”流程后，本项目做本地化改造：

- 不从电子税务局自动获取备案任务；
- 改为用户上传 Excel 备案计划；
- 软件根据备案计划生成单票业务；
- 每票业务自动生成固定单证目录；
- 用户按目录上传、补充、维护单证附件；
- 软件自动判断单证是否备齐；
- 支持备案完成、不申报退税、导出、备份、恢复、日志和权限管理。

第一版面向单机使用：同一台 Windows 电脑、同一套本地资料库、管理员和普通用户轮流登录，不考虑多人同时操作。

采用“工作台中心型”。

第一版 UI 采用 Apple 风格启发的极简设计语言。

设计目标：

- 白色为主色调；
- 大面积留白；
- 低饱和度辅助色；
- 弱边框、浅阴影；
- 圆角卡片；
- 清晰字体层级；
- 减少不必要的装饰线和复杂图标；
- 重要操作使用克制的蓝色强调；

- 表格保持安静、清爽，避免过重网格线。

界面观感应接近：

```text
干净
克制
轻量
专业
高留白
低噪音
```

注意：只借鉴 Apple 的极简设计气质，不使用 Apple 商标、Logo、系统图标或受保护素材。

左侧菜单包括：

- 工作台；

## 4. 备案计划设计

### 4.1 备案计划编号

备案计划编号直接使用：

```text
退税所属期-批次号
```

示例：

```text
2026-01-001
2026-01-002
2026-02-001
```

不额外设置显示名称。

### 4.2 退税所属期

退税所属期在上传备案计划时由界面选择，不放入 Excel。

格式示例：

```text
2026-01
```

含义：该计划对应某个月份退税的单证备案。

### 4.3 批次号

一个退税所属期可能有多次退税申报，因此增加批次号。

规则：

- 批次号统一三位数字；
- 默认格式为 `001`、`002`、`003`；
- 软件根据已有计划自动建议下一个批次号；
- 用户可以手动调整批次号；
- 同一个计划编号不能重复。

### 5.1 文件格式

使用 `.xlsx` 文件。

第一版只支持一个工作表。建议工作表名称为：

```text
备案计划
```

如果用户修改工作表名称，软件可读取第一个工作表，避免模板过于僵硬。

以下字段不放进 Excel，由上传界面填写：

- 退税所属期；
- 批次号。

软件自动生成：

- 计划编号。

- 退税所属期；
- 批次号；
- 计划编号；
- 单证状态；
- 备案状态；
- 是否不申报退税；
- 不申报原因；
- 附件信息。

这些字段由软件生成或由后续操作产生。

### 5.5 校验规则

导入前整份校验。

只要有一行错误，整份计划不导入。

校验内容：

- 报关单号不能为空；
- 出口日期不能为空；
- 报关单号在 Excel 内不能重复；
- 报关单号在资料库内不能已存在；
- 出口日期必须能识别为日期；
- FOB价美元如果填写，必须是数字；
- 申报退税额如果填写，必须是数字；
- 表头必须能匹配模板字段。

### 5.6 错误清单

校验失败时显示错误清单：

| Excel 行号 | 字段 | 错误原因 | 修改建议 |
|---|---|---|---|
| 5 | 报关单号 | 不能为空 | 补充报关单号 |
| 8 | 出口日期 | 日期格式无法识别 | 改为 2026-01-15 |
| 12 | 报关单号 | 与第 3 行重复 | 检查是否重复导入 |

用户修改 Excel 后重新上传。

### 5.7 成功导入后

导入成功后：

- 创建备案计划；
- 复制原始 Excel 到计划的 `source` 文件夹；
- 每行生成一票业务；
- 每票业务初始单证状态为“未备齐”；
- 每票业务初始备案状态为“未完成”；
- 写入操作日志；
- 跳转到该备案计划的业务列表。

## 7. 状态设计

### 7.1 单证状态

单证状态表示资料是否齐全。

取值：

- 未备齐；
- 已备齐；
- 不申报退税。

规则：

- 普通业务根据必备单证自动计算；
- 必备单证全部至少上传一份后，单证状态为“已备齐”；
- 任一必备单证缺失时，单证状态为“未备齐”；
- 非必备单证缺失不影响状态；
- 不申报退税业务不再检查缺少单证。

### 7.2 备案状态

备案状态表示是否完成备案确认。

取值：

- 未完成；
- 已完成；
- 不申报退税。

规则：

- 初始为“未完成”；
- 只有单证状态为“已备齐”时，才能标记为“已完成”；
- 如果未备齐时尝试完成备案，软件列出缺少项；
- 管理员转为“不申报退税”后，备案状态也变为“不申报退税”。

### 7.3 不申报退税

管理员可将业务转为“不申报退税”。

规则：

- 仅管理员可操作；
- 必须填写备注原因；
- 单证状态和备案状态均变为“不申报退税”；
- 不再检查必备单证；
- 已有附件继续保留；
- 操作写入日志。

管理员也可以恢复正常业务。

恢复规则：

- 仅管理员可操作；
- 取消不申报退税标记；
- 单证状态重新按必备单证计算；
- 备案状态恢复为“未完成”；
- 原备注保留，不自动清空；
- 操作写入日志。

## 8. 附件设计

### 8.1 上传规则

- 上传前必须先选择具体单证项；
- 不能直接上传到整票业务根目录；
- 可一次选择多个文件；
- 允许拖拽上传，但仍必须先选中单证项；
- 如果没有选中单证项，上传按钮不可用或提示“请先选择单证项”；
- 上传后文件复制到软件管理的本地资料库；
- 保留原始文件名；
- 同名文件自动追加序号，不覆盖旧文件；
- 同一单证项允许混合多种格式、多文件。

### 8.2 支持格式

软件内预览：

- PDF；
- JPG；
- JPEG；
- PNG。

调用本机默认程序打开：

- Word；
- Excel。

第一版不允许上传其他格式。

### 8.3 删除与替换

权限：

- 普通用户可以删除、替换自己上传的附件；
- 管理员可以删除、替换所有附件。

规则：

- 删除附件会删除资料库中的文件；
- 替换附件就是覆盖旧文件；
- 替换时保持附件所属单证项不变；
- 替换后的文件名使用新文件名；
- 替换文件格式不合法时拒绝替换；
- 如果替换失败，原文件必须保留不变；
- 删除和替换都写入操作日志。

## 10. 单票详情页设计

单票详情页是第一版核心页面。

### 10.1 页面结构

- 顶部：业务基础信息与状态；
- 左侧：固定单证目录树；
- 右侧：当前单证项附件列表与预览区；
- 底部：备注、缺少项、操作日志入口。

### 10.2 目录状态显示

- `✓`：已上传；
- `!`：必备但未上传；
- `-`：非必备且未上传；
- 数字：已有附件数量。

进入详情页时：

- 如果有缺少的必备项，默认选中第一个缺少项；
- 如果已备齐，默认选中第一个已有附件的单证项。

### 10.3 功能

- 编辑业务字段；
- 选择单证项；
- 上传多个附件；
- 删除附件；
- 替换附件；
- PDF / 图片软件内预览；
- Word / Excel 调用本机默认程序打开；
- 查看缺少项；
- 已备齐后标记备案完成；
- 管理员转为不申报退税；
- 管理员恢复正常业务。

## 11. 备案计划页设计

备案计划页用于管理批次计划。

功能：

- 上传 Excel 备案计划；
- 选择退税所属期；
- 自动建议批次号；
- 生成计划编号；
- 查看计划列表；
- 查看计划下业务数量、未备齐数量、已完成数量；
- 进入计划业务列表；
- 管理员删除空计划。

计划只有为空时才允许删除。

## 12. 删除规则

### 12.1 删除单票业务

仅管理员可操作。

单票删除规则：

- 二次确认；
- 如果业务已有附件，确认提示必须明确“会删除该票所有附件”；
- 删除后业务从列表消失；
- 业务本体和附件不保留；
- 操作日志记录删除摘要。

### 12.2 批量删除单票业务

仅管理员可操作。

规则：

- 只能在某个备案计划内批量删除；
- 必须先勾选业务；
- 二次确认；
- 确认信息显示将删除多少票业务、多少个附件；
- 删除后业务与附件不保留；
- 操作日志记录删除摘要。

### 12.3 删除备案计划

仅管理员可操作。

规则：

- 只有空计划可以删除；
- 如果计划下还有业务，不允许删除；
- 删除前二次确认；
- 操作日志记录。

“空计划”指该计划下没有任何单票业务。

## 13. 导出资料设计

### 13.1 单票导出

规则：

- 生成 `报关单号.zip`；
- ZIP 内直接平铺全部已上传源文件；
- 不保留单证分类文件夹；
- 不导出未上传项；
- 不创建空文件夹；
- 未备齐业务允许导出当前资料；
- 不生成备案信息表；
- 文件重名时自动追加序号，例如 `发票.pdf`、`发票 (2).pdf`。

### 13.2 整计划导出

规则：

- 选择备案计划；
- 创建以备案计划编号命名的文件夹；
- 文件夹内每票业务各生成一个 `报关单号.zip`；
- 每个 ZIP 内仍平铺该票已上传源文件。

## 14. 用户与权限

### 14.1 初始管理员

第一版初始账号：

```text
用户名：admin
密码：admin
角色：管理员
```

不强制首次登录修改密码。

软件应提示这是低安全性的初始密码，并保留管理员主动修改密码功能。

### 14.2 角色

管理员：

- 拥有全部普通用户能力；
- 新增、停用用户；
- 重置密码；
- 设置角色；
- 删除单票业务；
- 批量删除单票业务；
- 删除空备案计划；
- 转为不申报退税；
- 恢复正常业务；

### 15.2 可见性

## 21. 技术架构

采用：

```text
Electron + React + TypeScript + SQLite + 本地文件资料库
```

### 21.1 分层

```text
界面层 React
  ↓
业务接口层 Electron IPC
  ↓
本地服务层 Node.js
  ↓
SQLite 数据库 + 文件资料库
```

### 21.2 React 界面层

负责：

- 登录页；
- 工作台；
- 备案计划页；
- 单票详情页；
- 导出资料页；
- 备份与恢复页；
- 用户管理页；
- 操作日志页；
- 系统设置页。

界面层不直接读写硬盘文件。

### 21.3 Electron 主进程 / 本地服务层

负责：

- 读取 Excel 模板；
- 校验模板；
- 复制附件；
- 删除附件；
- 替换附件；
- 打开本机文件；
- 打包 ZIP；
- 备份恢复；
- 访问 SQLite 数据库。

### 21.4 SQLite 数据库

保存：

- 用户；
- 备案计划；
- 单票业务；
- 附件记录；
- 操作日志；
- 系统设置。

附件文件本体不放进数据库，放在资料库文件夹里。数据库只记录路径和元数据。

## 25. 第一版实施决策

为避免实施阶段产生歧义，第一版固定采用以下决策：

- Excel 模板使用第 5.3 节列出的中文全称表头；
- 报关单号只校验非空和全库唯一，不增加长度或字符格式限制；
- 新建用户和修改密码时要求密码至少 8 个字符，初始管理员 `admin / admin` 作为唯一兼容例外；
- ZIP 内重名文件使用 `文件名 (2).扩展名`、`文件名 (3).扩展名` 的顺序追加格式；
- 完整备份包扩展名使用 `.wintax-backup`；
- 软件名称使用 `WINtax`；
- 第一版使用项目自有的简洁文字图标，不使用第三方品牌图标。
